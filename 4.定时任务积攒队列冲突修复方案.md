# 定时任务积攒队列冲突修复方案

## 问题描述

### 问题1：定时任务积攒队列冲突（最优先级）
定时任务到时间后，进入积攒列表，然后显示"执行完毕"，但实际上任务没有执行，定时任务列表也被清空。

### 问题2：任务函数未注册（高优先级）
错误信息：`任务函数不存在: batchUseItems`
定时任务依赖的 `batchUseItems` 函数没有在 `originalTaskFunctionMap` 中注册，导致依赖验证失败，任务取消执行。

---

## 问题原因

### 问题1原因：三方冲突

项目中存在三个功能都在管理 `isRunning` 状态：

1. **分批执行** (`executeInBatches`)
   - 位置：`src/views/BatchDailyTasks.vue`
   - 功能：将大任务分成小批次执行
   - 管理：在任务开始时调用 `startTask()`，结束时调用 `stopTask()`

2. **积攒列表** (`checkAndExecuteQueuedTasks`)
   - 位置：`src/views/BatchDailyTasks.vue`
   - 功能：当任务冲突时，将任务加入队列等待执行
   - 管理：在积攒队列开始时调用 `startTask()`，结束时调用 `stopTask()`

3. **任务函数** (如 `batchLegacyClaim`)
   - 位置：`src/utils/batch/tasks*.js`
   - 功能：执行具体的任务逻辑
   - 管理：在任务函数内部调用 `startTask()` 和 `stopTask()`

**冲突流程：**
```
1. 定时任务到时间，检测到有其他任务在执行
2. 定时任务加入积攒队列
3. 积攒队列开始执行
4. 积攒队列调用 executeInBatches
5. executeInBatches 调用任务函数
6. 任务函数内部调用 stopTask()，将 isRunning 设为 false
7. executeInBatches 的 finally 块恢复 isRunning 为 true
8. 但任务函数的 stopTask() 同时设置了 shouldStop = true
9. 导致后续任务被跳过
10. 积攒队列执行器的 finally 块又调用 stopTask()
11. 页面刷新逻辑检测到 isRunning = false，触发页面刷新
12. 定时任务数据丢失
```

### 问题2原因：任务函数未注册

`batchUseItems` 函数在 `taskFunctionMap` 中已注册（用于UI按钮点击），但在 `originalTaskFunctionMap` 中未注册（用于定时任务依赖验证）。

**文件位置：** `src/views/BatchDailyTasks.vue`
**问题位置：** `originalTaskFunctionMap` 定义中缺少 `batchUseItems`

---

## 修复方案（按优先级排序）

### 方案一：短期方案（推荐先实施）

在 `executeInBatches` 中保存/恢复 `isRunning` 状态。

#### 修改文件

`src/views/BatchDailyTasks.vue`

#### 修改位置

`executeInBatches` 函数中，搜索以下代码：

```javascript
// 重置 shouldStop，确保任务可以正常执行
batchTaskStore.resetShouldStop();
try {
  await taskFunction();
} catch (error) {
  addLog({
    time: new Date().toLocaleTimeString(),
    message: `执行批次时出错: ${error.message}`,
    type: "error",
  });
  console.error('taskFunction error:', error);
}

// 重要：taskFunction 执行后可能会调用 stopTask() 将 shouldStop 设为 true
// 必须立即重置 shouldStop，否则下一批会跳过
batchTaskStore.resetShouldStop();
```

#### 替换为

```javascript
// 重置 shouldStop，确保任务可以正常执行
batchTaskStore.resetShouldStop();

// 重要：保存当前的 isRunning 状态
// 任务函数内部可能会调用 stopTask() 改变 isRunning 状态
const wasRunningBeforeTask = batchTaskStore.isRunning;

try {
  await taskFunction();
} catch (error) {
  addLog({
    time: new Date().toLocaleTimeString(),
    message: `执行批次时出错: ${error.message}`,
    type: "error",
  });
  console.error('taskFunction error:', error);
} finally {
  // 重要：恢复 isRunning 状态
  // 如果任务执行前 isRunning 为 true，但任务函数内部调用了 stopTask() 将其设为 false
  // 则需要恢复为 true，以确保分批执行和积攒队列的正常工作
  if (wasRunningBeforeTask && !batchTaskStore.isRunning) {
    batchTaskStore.startTask();
  }
}

// 重要：taskFunction 执行后可能会调用 stopTask() 将 shouldStop 设为 true
// 必须立即重置 shouldStop，否则下一批会跳过
batchTaskStore.resetShouldStop();
```

#### 关键点

1. 在调用 `taskFunction()` 之前，保存 `isRunning` 状态到 `wasRunningBeforeTask`
2. 在 `try-catch` 后添加 `finally` 块
3. 在 `finally` 块中，如果 `isRunning` 被任务函数改为 false，则恢复为 true

### 方案二：积攒队列执行器优化

移除任务执行过程中的 `stopTask()` 调用。

#### 修改文件

`src/views/BatchDailyTasks.vue`

#### 修改位置

`checkAndExecuteQueuedTasks` 函数中，搜索以下代码：

```javascript
} finally {
  // 清除任务执行标记
  if (task.id) {
    safeLocalStorage.removeItem(`task_executing_${task.id}`);
  }
  batchTaskStore.stopTask();
}
```

#### 替换为

```javascript
} finally {
  // 清除任务执行标记
  if (task.id) {
    safeLocalStorage.removeItem(`task_executing_${task.id}`);
  }
  // 移除 stopTask 调用，由积攒队列执行器的 finally 块统一管理
}
```

#### 关键点

1. 只移除任务执行过程中的 `stopTask()` 调用
2. 保留积攒队列执行器最后的 `stopTask()` 调用（在所有任务完成后）

### 方案三：长期方案（可选）

修改所有任务文件，移除内部的 `startTask/stopTask` 调用，让 `executeInBatches` 成为唯一管理 `isRunning` 状态的地方。

8. `src/utils/batch/tasksBlackMarket.js` - 待修改
9. `src/utils/batch/tasksStore.js` - 待修改
10. `src/utils/batch/tasksDungeon.js` - 待修改

#### 修改步骤

**步骤1：定位 startTask 调用**

在每个文件中搜索：
```javascript
if (startTask) startTask();
else { startTask(); }
```

替换为：
```javascript
// 移除 startTask 调用，由调用方管理 isRunning 状态
```

**步骤2：定位 stopTask 调用**

在每个文件中搜索：
```javascript
if (stopTask) stopTask();
else { stopTask(); }
```

替换为：
```javascript
// 移除 stopTask 调用，由调用方管理 isRunning 状态
```

**步骤3：添加注释说明**

在函数注释中添加：
```javascript
/**
 * 函数名
 * 注意：此函数不管理 isRunning 状态，由调用方（executeInBatches）管理
 */
```

#### 修改示例

**修改前：**
```javascript
const batchLegacyClaim = async () => {
  if (selectedTokens.value.length === 0) return;
  if (startTask) startTask();
  else { startTask(); }

  // ... 任务逻辑 ...

  if (stopTask) stopTask();
  else { stopTask(); }
};
```

**修改后：**
```javascript
/**
 * 批量领取功法残卷
 * 注意：此函数不管理 isRunning 状态，由调用方（executeInBatches）管理
 */
const batchLegacyClaim = async () => {
  if (selectedTokens.value.length === 0) return;
  // 移除 startTask 调用，由调用方管理 isRunning 状态

  // ... 任务逻辑 ...

  // 移除 stopTask 调用，由调用方管理 isRunning 状态
};
```

## 推荐修复顺序

**重要：按以下顺序逐步实施，每完成一步就测试！**

1. **第一步：实施方案一（短期方案）**
   - 修改 `executeInBatches` 函数
   - 测试是否有效

2. **第二步：实施方案二（积攒队列优化）**
   - 修改 `checkAndExecuteQueuedTasks` 函数
   - 测试是否有效

3. **第三步：实施方案三（长期方案）- 可选**
   - 如果前两步有效，可以跳过此步
   - 如果想代码更清晰，可以逐步修改任务文件

## 验证方法

### 1. 检查日志

修改后，日志中不应该再出现：
```
=== 检测到任务函数改变了 isRunning 状态，已恢复 ===
```

如果出现，说明该任务文件还没有修改。

### 2. 功能测试

测试以下场景：

1. **单独执行任务**
   - 选择几个账号
   - 点击"收残卷"按钮
   - 观察日志，任务应该正常执行完成

2. **积攒队列功能**
   - 先点击"收残卷"开始执行
   - 在收残卷执行过程中，点击"送残卷"
   - 观察日志：
     - 应该显示"检测到冲突，加入积攒列表等待"
     - 收残卷完成后，应该自动执行送残卷
     - 不应该出现任务重复加入积攒列表的情况

3. **定时任务**
   - 设置两个定时任务，时间间隔1-2分钟
   - 等待第一个任务执行
   - 在第一个任务执行时，第二个任务应该加入积攒队列
   - 第一个任务完成后，第二个任务应该自动执行
   - 定时任务列表不应该被清空

### 3. 代码检查

检查任务文件中是否还有：
- `startTask()` 调用
- `stopTask()` 调用

---

## 注意事项

1. **不要一次性修改所有文件**：建议逐个修改，每修改一个就测试一个
2. **保留注释**：修改后的注释有助于后续维护
3. **检查依赖**：确保任务函数不再依赖 `startTask/stopTask` 的副作用
4. **测试充分**：修改后必须测试单独执行和积攒队列执行两种场景

## 总结

**优先级：**
- 方案一 > 方案二 > 方案三
- 方案一和方案二可以立即解决问题
- 方案三是可选的，为了代码更清晰、更易维护

**核心思路：**
- 让 `executeInBatches` 成为唯一管理 `isRunning` 状态的地方
- 任务函数不再管理 `isRunning` 状态
- 积攒队列执行器只在所有任务完成后才调用 `stopTask()`
