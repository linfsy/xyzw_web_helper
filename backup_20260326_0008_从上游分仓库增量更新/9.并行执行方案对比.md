# 并行执行功能 - 方案对比文档

## 用户需求

| 开关状态 | 场景 | 期望行为 |
|---------|------|---------|
| 开启 ✅ | 当前任务执行中，下个任务到时间 | 加入积攒队列，等待当前任务完成后**依次执行** |
| 关闭 ❌ | 当前任务执行中，**多个任务到时间** | **所有到时间的任务立即一起执行**（并行） |

**关键点**：开关关闭时，不管到时间多少个任务，都是一起执行。

---

## 方案1：修改冲突检测逻辑（简单方案）

### 实现方式

修改 `startScheduler` 函数中的冲突检测逻辑，当开关关闭时跳过 `isRunning` 检查。

### 代码修改

**修改前**：
```javascript
} else if (batchTaskStore.isRunning) {
  if (batchSettings.enableQueueOnConflict) {
    // 加入积攒队列
    continue;
  } else {
    // 记录日志，不continue
  }
}
```

**修改后**：
```javascript
} else if (batchTaskStore.isRunning && batchSettings.enableQueueOnConflict) {
  // 只有当开关开启时才检查 isRunning
  // 加入积攒队列
  continue;
} else if (batchTaskStore.isRunning && !batchSettings.enableQueueOnConflict) {
  // 开关关闭，有任务运行中，记录日志
  // 不continue，继续执行
}
```

### 优点

1. ✅ **实现简单**：只需要修改冲突检测的条件判断
2. ✅ **改动量小**：只修改2个位置的 `if` 条件
3. ✅ **符合直觉**：开关关闭时直接执行，不检查 `isRunning`
4. ✅ **向后兼容**：不影响现有的积攒队列功能

### 缺点

1. ❌ **`isRunning` 状态混乱**：
   - 多个任务同时设置 `isRunning = true`
   - 第一个任务完成后调用 `stopTask()`，设置 `isRunning = false`
   - 此时其他任务还在执行，但 `isRunning` 已经是 `false`
   - 新任务检测时认为没有任务运行

2. ❌ **`saveRunningTask` 覆盖问题**：
   - 多个任务同时执行，最后一个任务会覆盖前面的任务信息
   - 刷新页面后只能恢复最后一个任务

3. ❌ **`checkAndExecuteQueuedTasks` 多次调用**：
   - 多个任务同时完成时会多次调用
   - 可能导致积攒队列执行混乱

4. ❌ **依赖检测时机**：
   - 任务2启动后，`isRunning = true`
   - 任务3检测时，如果任务2的 `startTask()` 还没执行完，`isRunning` 还是 `false`
   - 任务3可能不会检测到冲突，导致3个任务同时执行

### 执行流程

**场景**：当前任务执行中，3个定时任务同时到时间

```
时间轴：
0s: 任务1（运行中）
1s: 任务2检测 → isRunning=true → 记录"将并行执行" → 启动任务2
2s: 任务3检测 → isRunning=true（任务2的startTask已执行）→ 记录"将并行执行" → 启动任务3
3s: 任务1完成 → stopTask() → isRunning=false
4s: 任务2完成 → stopTask() → isRunning=false
5s: 任务3完成 → stopTask() → isRunning=false
```

**问题**：任务2和任务3在任务1完成前就启动了，但 `isRunning` 状态在任务1完成后才重置。

## 方案3：收集任务后统一执行（推荐方案）

### 实现方式

将任务收集和执行分离，先收集所有需要执行的任务，然后根据开关决定执行方式。

### 代码修改

**修改前**：
```javascript
// 依次执行定时任务（不是并行）
for (const task of tasksToRun) {
  // ... 检查逻辑 ...
  if (shouldRun) {
    // ... 冲突检测 ...
    // 执行任务
  }
}
```

**修改后**：
```javascript
// 第一阶段：收集需要执行的任务
const tasksToExecute = [];

for (const task of tasksToRun) {
  if (shouldRun) {
    // ... 检查逻辑 ...

    if (batchSettings.enableQueueOnConflict && batchTaskStore.isRunning) {
      // 开关开启，加入积攒队列
      batchTaskStore.addToTaskQueue({...});
    } else {
      // 开关关闭 或 无冲突，收集任务
      tasksToExecute.push(task);
    }
  }
}

// 第二阶段：执行收集到的任务
if (tasksToExecute.length > 0) {
  if (batchSettings.enableQueueOnConflict) {
    // 顺序执行
    for (const task of tasksToExecute) {
      await executeScheduledTask(task);
    }
  } else {
    // 并行执行
    addLog({ message: `本次有${tasksToExecute.length}个任务到时间，将一起并行执行` });
    for (const task of tasksToExecute) {
      executeScheduledTask(task).then(...);
    }
  }
}
```

### 优点

1. ✅ **逻辑清晰**：收集 → 执行，两阶段分离
2. ✅ **符合调度策略**：开关控制调度方式（串行/并行）
3. ✅ **避免状态混乱**：不依赖全局 `isRunning` 状态
4. ✅ **用户体验好**：可以添加日志显示"本次有X个任务到时间"
5. ✅ **可扩展性强**：以后可以添加"最多并行3个任务"等功能
6. ✅ **任务独立**：每个任务独立管理状态，不互相干扰

### 缺点

1. ❌ **改动量大**：需要重构 `startScheduler` 函数的核心逻辑
2. ❌ **测试复杂**：需要测试多种场景（单任务、多任务、开关开启/关闭）

### 执行流程

**场景**：当前任务执行中，3个定时任务同时到时间

```
时间轴：
0s: 收集阶段 → 检测到3个任务到时间
1s: 执行阶段 → 并行启动任务2、任务3
2s: 任务2、任务3 执行中
3s: 任务1完成
4s: 任务2完成
5s: 任务3完成
```

**优势**：所有到时间的任务同时启动，真正实现并行执行。

## 推荐方案

**推荐：方案3**

### 原因

1. **完全符合用户需求**：
   - "不管到时间多少个任务都是一起执行"
   - 方案3天然支持：收集 → 一起执行

2. **避免状态管理问题**：
   - 不依赖全局 `isRunning` 状态
   - 每个任务独立管理自己的状态

3. **更好的用户体验**：
   - 可以添加日志："本次有3个任务到时间，将一起并行执行"
   - 用户可以清楚知道有多少个任务同时执行

4. **可扩展性强**：
   - 以后可以添加"最多并行3个任务"等功能
   - 可以添加"任务优先级"等功能

5. **长期维护性**：
   - 逻辑清晰，易于理解和维护
   - 减少状态管理问题的出现

## 测试建议

1. 创建2个定时任务：
   - 任务A：10:00:00 执行
   - 任务B：10:00:01 执行

2. 关闭"冲突时加入积攒队列"开关

3. 观察日志：
