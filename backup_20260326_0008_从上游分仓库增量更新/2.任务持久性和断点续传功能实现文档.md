### 1.2 什么是任务持久性和断点续传？

**任务持久性**：
- 即使页面刷新或应用重启，任务状态也能保存和恢复
- 任务执行进度不会丢失
- 用户可以随时查看任务执行历史

**断点续传**：
- 任务执行过程中断（如网络中断、页面刷新）后，可以从中断点继续执行
- 不需要从头开始重新执行所有任务
- 节省时间和资源

### 1.3 为什么需要这个功能？

**场景1：页面意外刷新**
- 用户在执行大量账号任务时，不小心刷新了页面
- 没有持久化：所有进度丢失，需要重新开始
- 有持久化：自动恢复进度，继续执行

**场景2：网络中断**
- 任务执行过程中网络不稳定，导致连接断开
- 没有持久化：任务失败，需要手动重新开始
- 有持久化：网络恢复后自动继续执行

**场景3：应用重启**
- 手机内存不足，APK被系统关闭
- 没有持久化：任务完全丢失
- 有持久化：重启后自动恢复并继续执行

## 三、核心功能实现

### 3.1 任务状态存储

#### 3.1.1 存储的数据结构

```typescript
// 任务状态接口
interface TaskState {
  // 任务基本信息
  isRunning: boolean              // 是否正在运行
  taskName: string               // 任务名称
  timestamp: number              // 保存时间戳
  
  // 进度信息
  currentProgress: number         // 当前进度（百分比）
  currentBatchIndex: number       // 当前批次索引
  totalBatches: number            // 总批次数
  
  // 账号信息
  selectedTokens: string[]        // 选中的账号ID列表
  completedTokens: string[]       // 已完成账号ID列表
  currentRunningTokenId: string | null  // 当前正在执行的账号ID
  
  // 任务配置
  selectedTasks: string[]         // 选中的任务列表
  taskQueue: any[]                // 积攒的任务队列
}

// 运行中任务信息接口
interface RunningTaskInfo {
  id: number                      // 任务ID
  name: string                    // 任务名称
  selectedTokens: string[]        // 选中的账号
  selectedTasks: string[]         // 选中的任务
  timestamp: number               // 开始时间
}
```

#### 3.1.2 存储位置

| 键名 | 用途 | 过期时间 |
|------|------|----------|
| `batch_task_state` | 任务执行状态（进度、批次等） | 24小时 |
| `running_task_info` | 运行中任务信息 | 5分钟 |
| `executingState` | 执行状态标记 | 页面刷新后清除 |
| `batch_task_queue` | 积攒任务队列 | 手动清除 |

#### 3.1.3 代码实现

```typescript
// src/stores/batchTaskStore.ts

const TASK_STATE_KEY = 'batch_task_state'

/**
 * 保存任务状态到 localStorage
 */
function saveTaskState(state: Partial<TaskState>) {
  const existingState = getTaskState()
  const newState = {
    ...existingState,
    ...state,
    timestamp: Date.now(),
  }
  localStorage.setItem(TASK_STATE_KEY, JSON.stringify(newState))
}

/**
 * 从 localStorage 获取任务状态
 */
function getTaskState(): TaskState | null {
  try {
    const saved = localStorage.getItem(TASK_STATE_KEY)
    if (saved) {
      return JSON.parse(saved)
    }
  } catch (e) {
    console.error('获取任务状态失败:', e)
  }
  return null
}

/**
 * 清除任务状态
 */
function clearTaskState() {
  localStorage.removeItem(TASK_STATE_KEY)
}

/**
 * 检查是否有未完成的任务
 */
function hasUnfinishedTask(): boolean {
  const state = getTaskState()
  if (!state) return false
  
  // 检查任务是否过期（超过24小时视为过期）
  const now = Date.now()
  const taskAge = now - (state.timestamp || 0)
  if (taskAge > 24 * 60 * 60 * 1000) {
    clearTaskState()
    return false
  }
  
  // 检查是否有正在运行的任务且有选中的账号
  return state.isRunning === true && 
         state.selectedTokens && 
         state.selectedTokens.length > 0
}
```

### 3.2 任务进度保存

#### 3.2.1 何时保存进度？

| 时机 | 保存内容 | 原因 |
|------|----------|------|
| 任务开始时 | 初始状态 | 记录任务开始，便于恢复 |
| 每个批次完成时 | 当前批次索引、已完成账号 | 实现断点续传 |
| 任务暂停时 | 当前进度、状态 | 用户主动暂停，保存状态 |
| 页面刷新前 | 所有状态 | 防止意外丢失 |

#### 3.2.2 代码实现

```typescript
// 在 executeInBatches 函数中

// 1. 任务开始时保存初始状态
batchTaskStore.saveTaskState({
  isRunning: true,
  taskName,
  selectedTokens: selectedTokens.value,
  selectedTasks: selectedTasks.value,
  totalBatches,
  currentBatchIndex: 0,
  currentProgress: 0,
})

// 2. 每个批次完成后保存进度
for (let batchIndex = startBatchIndex; batchIndex < totalBatches; batchIndex++) {
  // ... 执行批次任务 ...
  
  // 保存进度
  batchTaskStore.saveTaskState({
    currentBatchIndex: batchIndex + 1,
    currentProgress: Math.round(((batchIndex + 1) / totalBatches) * 100),
  })
}

// 3. 任务完成后清除状态
batchTaskStore.clearTaskState()
```

### 3.3 断点续传实现

#### 3.3.1 简化版实现（当前采用）

**设计思路**：
- 不记录具体执行到哪个账号
- 只记录任务配置（选中的任务、账号列表）
- 页面刷新后，重新执行整个任务
- 优点：简单、可靠、易于维护
- 缺点：不能精确断点续传，会重复执行已完成的批次

**代码实现**：

```typescript
// src/views/BatchDailyTasks.vue

// 在 onMounted 中检查是否有运行中被刷新的任务
onMounted(async () => {
  // ... 其他初始化代码 ...
  
  // 检测运行中被刷新的任务
  const savedRunningTask = safeLocalStorage.getItem(RUNNING_TASK_KEY)
  if (savedRunningTask) {
    try {
      const task = JSON.parse(savedRunningTask)
      // 检查是否在5分钟内（避免很久以前的任务被重新执行）
      if (Date.now() - task.timestamp < 5 * 60 * 1000) {
        addLog({
          time: new Date().toLocaleTimeString(),
          message: `=== 检测到运行中被刷新的任务：${task.name}，3秒后重新执行 ===`,
          type: "warning",
        })
        
        // 延迟3秒后重新执行
        await new Promise(r => setTimeout(r, 3000))
        
        // 重新执行整个任务（简化版，不做断点续传）
        await manualExecuteTask(task)
      }
      
      // 清除已处理的任务
      clearRunningTask()
    } catch (e) {
      console.error('恢复运行中任务失败:', e)
      clearRunningTask()
    }
  }
})
```

#### 3.3.2 完整版断点续传（未来可考虑）

**设计思路**：
- 记录每个账号的执行状态（未开始/进行中/已完成/失败）
- 页面刷新后，只执行未完成的账号
- 优点：精确断点续传，不会重复执行
- 缺点：复杂、需要维护更多状态

**实现难点**：
1. **状态一致性**：如何确保保存的状态与实际执行状态一致
2. **错误处理**：某个账号失败后，如何记录和处理
3. **并发控制**：多个任务同时执行时的状态管理
4. **存储空间**：大量账号的状态信息可能占用较多存储空间

**建议**：
- 当前简化版已满足大部分需求
- 如果未来需要精确断点续传，可以考虑完整版实现
- 或者采用折中方案：记录批次级别的进度，不记录账号级别的进度

## 四、积攒任务队列

### 4.1 什么是积攒任务队列？

当多个任务同时触发或任务执行期间有新任务加入时，这些任务会被放入积攒任务队列，按顺序依次执行。

**典型场景**：
1. 手动执行任务时，定时任务到时间
2. 任务执行期间，暂停时间结束
3. 多个定时任务同时到时间

### 4.2 积攒任务队列的实现

#### 4.2.1 数据结构

```typescript
interface QueuedTask {
  id?: number                    // 任务ID
  name?: string                  // 任务名称
  taskName?: string              // 任务标识
  selectedTokens?: string[]      // 选中的账号
  selectedTasks?: string[]       // 选中的任务
  runType?: string               // 运行类型
  params?: any                   // 额外参数
}
```

#### 4.2.2 核心逻辑

```typescript
// 添加任务到积攒队列
const addToTaskQueue = (task: QueuedTask) => {
  batchTaskStore.addToTaskQueue(task)
  saveTaskQueueToStorage()  // 保存到 localStorage
}

// 检查并执行积攒队列
const checkAndExecuteQueuedTasks = async () => {
  const queue = batchTaskStore.taskQueue
  if (queue.length === 0) return
  
  // 防止重复执行
  if (isExecutingQueuedTasks.value) return
  isExecutingQueuedTasks.value = true
  
  try {
    // 依次执行队列中的任务
    for (const task of queue) {
      await executeTask(task)
      
      // 执行完成后从队列中移除
      removeFromTaskQueue(task.id)
    }
  } finally {
    isExecutingQueuedTasks.value = false
  }
}

// 任务执行完成后，自动检查并执行积攒队列
const executeInBatches = async (...) => {
  // ... 执行任务 ...
  
  // 任务完成后检查积攒队列
  if (!isFromQueue) {
    checkAndExecuteQueuedTasks()
  }
}
```

### 4.3 积攒任务的持久化

```typescript
// 保存积攒队列到 localStorage
const saveTaskQueueToStorage = () => {
  safeLocalStorage.setItem('batch_task_queue', 
    JSON.stringify(batchTaskStore.taskQueue))
}

// 从 localStorage 加载积攒队列
const loadTaskQueueFromStorage = () => {
  const savedQueue = safeLocalStorage.getItem('batch_task_queue')
  if (savedQueue) {
    try {
      const queue = JSON.parse(savedQueue)
      queue.forEach(task => {
        batchTaskStore.addToTaskQueue(task)
      })
    } catch (e) {
      console.error('加载积攒队列失败:', e)
    }
  }
}
```

## 六、常见问题排查

### 6.1 任务不恢复

**现象**：页面刷新后任务没有自动恢复

**排查步骤**：
1. 检查浏览器控制台是否有错误信息
2. 检查 localStorage 中是否有 `running_task_info`
3. 检查时间戳是否在5分钟内
4. 检查 `manualExecuteTask` 函数是否正常工作

**可能原因**：
- 超过5分钟，任务被视为过期
- `RUNNING_TASK_KEY` 不一致
- `manualExecuteTask` 函数执行失败

### 6.2 进度不保存

**现象**：任务执行过程中进度没有更新

**排查步骤**：
1. 检查 `saveTaskState` 是否被调用
2. 检查 localStorage 中 `batch_task_state` 是否更新
3. 检查是否有错误阻止保存

**可能原因**：
- `saveTaskState` 调用被注释掉
- localStorage 存储空间不足
- 序列化错误

### 6.3 积攒任务不执行

**现象**：有积攒任务但当前任务完成后没有自动执行

**排查步骤**：
1. 检查 `checkAndExecuteQueuedTasks` 是否被调用
2. 检查 `isExecutingQueuedTasks` 标志位
3. 检查积攒队列中是否有任务

**可能原因**：
- `isFromQueue` 参数传递错误
- `isExecutingQueuedTasks` 没有正确重置
- 积攒队列为空或格式错误

## 七、最佳实践

### 7.1 开发建议

1. **及时保存进度**：在每个关键节点保存进度，不要等太久
2. **简化状态**：只保存必要的状态，避免存储过多数据
3. **错误处理**：所有存储操作都要有错误处理
4. **过期处理**：设置合理的过期时间，避免旧数据干扰
5. **日志记录**：关键操作要记录日志，便于排查问题

### 7.2 使用建议

1. **不要频繁刷新**：虽然支持恢复，但频繁刷新会影响任务执行
2. **检查积攒队列**：定期查看积攒队列，确保任务按预期执行
3. **监控执行日志**：关注执行日志，及时发现异常
4. **合理设置定时任务**：避免设置过多同时执行的定时任务
